2026.07.10 · Espanol coverage: the whole interface, not just the chrome

Para usted — Phase 2 of Spanish support: the entire visible interface now translates when you switch to Espanol. Phase 1 covered tabs, buttons, and short labels (98 keys). Phase 2 adds the empty states, search placeholders, panel headings, the Today's Edition strip, alert builder labels and parameters, loading messages, and all control labels across every lens (Money, People, Land, Property, Rules, Meetings, Alerts) -- growing the dictionary from 98 to over 200 keys. A new residual-English coverage gate in the test suite verifies 15 high-visibility sentinel strings are absent in Espanol mode.
  • Dictionary grows from 98 to 200+ keys; all new keys carry full en + es translations.
  • Static HTML: data-i18n attributes on all translatable labels, options, buttons, and headings. New data-i18n-placeholder attribute pattern for search inputs.
  • JavaScript: hardcoded English strings in renderToday, loading states, empty states, and dynamic label assignments replaced with t() calls.
  • Test spec 12_language.py extended with a residual-English sentinel gate: 15 high-visibility phrases must be absent in Espanol mode.
  • Machine-translated; pending native review through community partners.

2026.07.10 · Spanish support + style-guide copy pass

For you — A language switcher now appears in the header (English / Espanol). Choosing Spanish translates all tabs, chips, and messages in the UI. Notices themselves stay in English, which is the official language of the City Record. Your preference is remembered across visits. Separately, time chips, deadline chips, and the feedback-category selector were updated to follow the NYC Web Content Style Guide: "9 a.m." (not "9 AM"), spelled-out numbers ("closes in two days"), and acronym expansions on first use (RFP, M/WBE, ZAP). City Record content is now marked translate="no" so machine-translation tools leave it intact.
  • Language switcher (English / Espanol) with USWDS-pattern native-language labels, stored in localStorage, and aria-pressed for keyboard users.
  • Runtime dictionary (i18n.js) with 98 keys in English and Spanish — no build step, plain script tag. Spanish is machine-translated and pending native review through community partners.
  • Confirmation and digest emails deliver in the subscriber's chosen language.
  • translate="no" on all notice-data containers; lang attribute on all language-labeled spans and UI chrome.
  • Key parity lint added to CI: a missing translation key fails the build.

2026.07.10 · Accessibility: an enforced floor, not a promise

For you — If you use a keyboard or a screen reader, the rough edges are getting fixed for real: the feedback form's category picker now works without a mouse, the plain-English search box announces itself properly, low-contrast text is corrected site-wide, and the "minimum award" filter genuinely disables when it doesn't apply instead of just fading. From now on, an automated accessibility check (axe) runs against every page in our test harness — a change that breaks accessibility fails the build. CONTRIBUTING and SECURITY were also rewritten to describe how the project is actually governed and defended.
  • The gate immediately earned its keep: it caught a third contrast violation (chip count badges at 55% opacity, ratio 3.2:1) that two human passes had missed. All six public pages now pass with zero critical/serious violations.
  • Per the NYC Web Content Style Guide we're aligning to (WCAG 2.2 + plain language) — more of that standard becomes automated checks next.

2026.07.10 · Three new front doors: email-in, MCP, and The Data

For you — Three new ways in. Subscribe by email: write to subscribe@crol-list.org in plain English ("construction awards over $500k") and you'll get a confirmation link back — no form, no CAPTCHA, just your words. For AI assistants: point any MCP client at api.crol-list.org/mcp to search notices and set up watches programmatically (double opt-in still applies — nothing sends without the address confirming). The Data: a new page showing the City Record at a glance — what's actually in it, publication volume, procurement mix, top agencies and vendors by cleaned dollars — computed live in your browser from NYC Open Data.
  • Inbound email → LLM-parsed watch with the same double-opt-in confirm flow as the site; hard daily ceilings and per-sender limits on every model-backed path, failing closed.
  • MCP tools: search_notices, get_notice (the daily-refreshed mirror), preview_watch, create_watch — documented on the API page.
  • Plain-English money searches understand more: amount ceilings ("under $500k") and procurement categories ("construction", "human services").
  • Digest delivery moved to a queue: each subscriber's digest retries independently, so one bad address can't block a run; daily send caps unchanged. (Groundwork from Dev Doshi's crol-alert.)

2026.07.10 · Honest data rules + a faster backbone (with Dev Doshi)

For you — Money filters and digests can no longer be hijacked by the dataset's data-entry errors: amounts of $10 billion or more (there's a $96 trillion typo in the official record) are excluded, while real multi-billion awards now correctly appear — the old cutoff silently dropped everything above $5 billion, including the largest legitimate award (about $6.68 billion). Pre-qualified-list notices with placeholder year-2090 dates now say "no fixed deadline (rolling)" instead of a date no one should calendar. The About page documents the dataset's quirks — what the City Record actually contains and how we correct for it.
  • Data-honesty rules grounded in Dev Doshi's exploratory analysis of the full 1.09M-row dataset (87.5% personnel-change skew, a data-entry-error amount tail, rolling-deadline placeholders, dual agency-name conventions) — thanks Dev.
  • New server-side mirror of recent notices (Cloudflare D1), refreshed by the daily cron from NYC Open Data — the source of truth is unchanged and site searches still query it live; the mirror makes alert matching and upcoming features fast, and stores each raw source row so upstream schema changes are recoverable.
  • Groundwork from the same analysis: a shared notice-search engine over the mirror (AND-of-ORs keyword matching with the honesty rules enforced in SQL), powering the next round of subscription features.
  • Continuous integration now runs both unit-test layers on every change (previous entry's multi-watch regression suite included).

2026.07.02 · Fix: vendors with punctuated names resolve again

For you — Vendor pages and vendor watches now work for names like "Leon D. Dematteis Construction Corp." Before this fix, clicking such a vendor showed "no awards on record" and a watch on them matched nothing — despite their awards being right there.
  • Root cause: entity resolution normalizes a name to a punctuation-free stem, but the server-side lookup prefix-matched that stem against the raw vendor_name column — so any vendor with mid-name punctuation could never match its own stem. Fixed by switching the lookup to Socrata full-text $q (punctuation-insensitive on both sides); the exact-stem filter still guarantees precision. Same fix in the site, the worker's entity watches, and the batch cross-reference tool.
  • Caught by functional spec 05; a new unit test pins the invariant (a vendor's own row must satisfy the compiled query).
